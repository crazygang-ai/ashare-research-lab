# UI Workbench Design

## Context

`ashare-research-lab` is a local offline A-share research system. Today it is mainly operated through CLI commands and generated Markdown / CSV artifacts. The first web UI should make the existing workflow easier to operate and review without changing the project boundary: it remains a research review tool, not a trading system.

The UI must preserve these statements throughout the product:

- `candidate list is not a trading instruction`
- `composite score is not a trading instruction`
- `backtest is a historical simulation, not a performance promise`
- `stock report is for research review only`

## Chosen Direction

Use a mixed research workbench:

- Default experience: review existing daily reports, stock reports, candidates, scores, and artifacts.
- Controlled run experience: allow the user to trigger single-stock report generation and the full HS300 daily workflow.
- Full workflow execution is advanced functionality, must be explicitly confirmed, and must display research-only caveats before launch.

The selected information architecture is `Review + Controlled Run`: the UI starts from research review and keeps execution controls in clearly marked `Runs` areas rather than presenting the app as a trading terminal.

## Architecture

The first version uses two local services:

- Backend: existing FastAPI service started by `ashare serve`, expected at `127.0.0.1:8008`.
- Frontend: new React / Vite / TypeScript app in `frontend/`, expected at `127.0.0.1:5173`.
- Frontend API access: Vite development proxy routes `/api` requests to FastAPI.
- Research storage: existing DuckDB files remain the source for research data.
- UI task history: JSON files under `data/service/workflow-runs/`.
- UI task logs: log files under `data/service/workflow-logs/<ui_run_id>/`.

The frontend does not write DuckDB directly. It calls backend APIs, and the backend invokes existing CLI commands or scripts. Research artifacts continue to be generated by the existing CLI/report/audit system, including `research_runs`, `research_artifacts`, and `run_manifest.json`.

## Backend API Additions

Keep existing `/api/v1/*` endpoints and add UI-specific endpoints under `/api/v1/ui/*`.

Required endpoints:

- `GET /api/v1/ui/config`
  Returns frontend configuration, API status, default DB path, report roots, safety flags, and research-only copy.
- `POST /api/v1/ui/runs/stock-report`
  Starts a single-stock `ashare stock-report` task.
- `POST /api/v1/ui/runs/hs300-daily`
  Starts the full `scripts/run_hs300_daily_research.sh` workflow.
- `GET /api/v1/ui/runs`
  Lists UI-triggered task history from local JSON logs.
- `GET /api/v1/ui/runs/{ui_run_id}`
  Returns task parameters, status, steps, output paths, and error summary.
- `GET /api/v1/ui/runs/{ui_run_id}/logs/stream`
  Streams stdout/stderr events for a running or recently completed task.

The existing endpoints remain important:

- `/api/v1/status`
- `/api/v1/artifacts`
- `/api/v1/runs`
- `/api/v1/scans/latest`
- `/api/v1/scoring/latest`
- `/api/v1/reports/daily/latest`
- `/api/v1/reports/stocks/latest`
- `/api/v1/reports/{artifact_id}/markdown`
- `/api/v1/stocks/{stock_code}/factors`

## Task Runner Design

Add a backend module such as `src/ashare/service/ui_runs.py`.

Responsibilities:

- Validate UI run parameters.
- Build explicit command arrays.
- Start subprocesses with `shell=False`.
- Capture stdout and stderr in real time.
- Write line-oriented logs to `data/service/workflow-logs/<ui_run_id>/`.
- Write task state JSON to `data/service/workflow-runs/<ui_run_id>.json`.
- Publish live log events through Server-Sent Events.
- Mark task step status as `queued`, `running`, `success`, `failed`, or `cancelled`.

Concurrency policy:

- First version allows only one mutating UI task at a time.
- Read-only artifact and report queries remain concurrent.
- If a task is already running, new run requests return `409 workflow_already_running`.
- Both HS300 daily workflow and single-stock report tasks share this lock to avoid DuckDB write conflicts.

Task history JSON should include:

- `ui_run_id`
- `task_type`
- `created_at`
- `started_at`
- `finished_at`
- `status`
- `params`
- `command_preview`
- `steps`
- `log_paths`
- `artifact_paths`
- `error_code`
- `error_message`

## Run Flows

### Single-Stock Report

User inputs:

- `stock_code`
- `as_of`
- `source_run_id`
- `score_run_id`
- optional `scan_run_id`
- `db_path`
- `output_dir`
- `run_id`

The UI may offer latest run suggestions from artifact metadata, but it must show the selected source and report paths before execution.

The backend command is an explicit `conda run -n ashare-research-lab ashare stock-report ...` invocation. On success, the UI links to the generated `stock_report.md`, `stock_factor_values.csv`, `stock_score_breakdown.csv`, `stock_metadata.json`, and `run_manifest.json`.

### HS300 Daily Workflow

User inputs:

- `as_of`
- `stock_code`
- optional `watchlist_file`
- optional `max_symbols`
- `cache_mode`
- optional environment overrides such as `DB`, `SOURCE`, `REPORT_ROOT`, and `SCORING_CONFIG`

The default backend command is:

```bash
scripts/run_hs300_daily_research.sh --as-of <DATE> --stock-code <CODE>
```

The UI displays the workflow steps:

1. `ingest`
2. `as-of`
3. `calculate-factors`
4. `factor-validation`
5. `scan`
6. `score`
7. `daily-report`
8. `stock-report`
9. optional watchlist stock reports

Before running, the UI must require confirmation of:

- AkShare HS300 members are a current snapshot, not strict historical PIT.
- `--max-symbols` is only for smoke / debug.
- Candidate lists and composite scores are research outputs, not trading instructions.
- Single-stock reports are for research review only.

## Pages

### Today

Default page for daily review.

Shows:

- API connection status.
- DB path and report root.
- Latest daily report summary.
- Latest data quality gate status.
- Latest scan and score counts.
- Latest stock report shortcut.
- Folded advanced entry for `Run HS300 Daily`.

### Stocks

Single-stock research page.

Shows:

- Stock code and as-of form.
- Factor values table.
- Score breakdown table.
- Candidate / scoring membership status.
- Latest stock report Markdown.
- `Generate Stock Report` action with command preview.

### Reports

Report reader.

Shows:

- Filterable report list by kind.
- Markdown viewer.
- CSV attachment tables.
- Links to source artifact paths and manifests.

Supported kinds include `daily_report`, `stock_report`, `factor_validation`, `scoring`, and `backtest`.

### Runs

Task center.

Shows:

- UI-triggered task history.
- Status badges.
- Step timeline.
- Real-time stdout/stderr log stream.
- Input parameters.
- Output artifacts.
- Failure reason and copyable command.

The full workflow is one-click by default, with step details expanded for diagnosis. Failed steps can expose a single-step rerun command preview, but automatic partial rerun can be deferred until after MVP.

### Artifacts

Artifact registry browser.

Shows:

- `run_id`
- artifact kind
- role
- path
- row count
- sha256
- created timestamp
- metadata

Supports filtering by kind, run id, date, and free text.

### Settings

Local UI configuration page.

Shows:

- API base URL.
- DB path.
- report roots.
- default `cache_mode`.
- default scoring config.
- workflow run permission state.
- UI preferences stored in browser local storage.

The first version should not edit core research configuration files from the browser. Durable research config changes still happen in `configs/*.yaml`.

## Visual And Interaction Guidelines

Use a restrained research tool style:

- Dense but organized information layout.
- Left sidebar navigation.
- Top bar with API status and latest run status.
- Tables for artifacts, candidates, scores, and factors.
- Clear pass / warning / fail status badges.
- One primary action per screen.
- Mutating actions visually distinct from read-only actions.
- No hero sections, decorative cards, or marketing-style layouts.
- No buy, sell, target price, position, or execution language.

Accessibility requirements:

- Keyboard navigation for forms, tables, and log panes.
- Visible focus states.
- 44px minimum clickable targets for primary controls.
- Color must not be the only indicator of pass / warning / fail.
- Body text no smaller than 16px where practical.
- Log stream must pause auto-scroll when the user scrolls upward.

## Error Handling

Backend errors should be structured with:

- `error_code`
- `message`
- `next_action`
- optional `details`

Important error categories:

- `invalid_params`
- `database_not_found`
- `workflow_http_disabled`
- `workflow_already_running`
- `provider_capability_failed`
- `command_failed`
- `artifact_not_found`
- `log_stream_disconnected`

The UI should show a concise error summary and a practical next step, such as checking logs, copying the command to terminal, checking `--data-source`, or running AkShare capability smoke.

## Configuration

Extend `configs/service.yaml` with a UI runner section:

```yaml
ui_runner:
  enabled: false
  max_concurrent_runs: 1
  history_dir: data/service/workflow-runs
  log_dir: data/service/workflow-logs
  require_confirmation: true
  allowed_commands:
    - stock-report
    - hs300-daily
```

The default should be safe. Running tasks from HTTP should remain disabled unless explicitly enabled in local configuration.

## Testing

Backend tests:

- UI task creation validates parameters.
- Disabled runner rejects mutating requests.
- Concurrent mutating task returns `409`.
- Task history JSON is written.
- Logs are written incrementally.
- SSE stream emits step and log events.
- Failed subprocess marks the task failed and stores stderr.

API tests:

- `POST /api/v1/ui/runs/stock-report`
- `POST /api/v1/ui/runs/hs300-daily`
- `GET /api/v1/ui/runs`
- `GET /api/v1/ui/runs/{ui_run_id}`
- `GET /api/v1/ui/runs/{ui_run_id}/logs/stream`

Frontend tests:

- Route rendering.
- API error display.
- Run form validation.
- Confirmation panel blocks unconfirmed execution.
- Run timeline renders step status.
- Log stream appends events.
- Artifact table filters and sorts.

End-to-end smoke:

- Start FastAPI with a mock UI task runner.
- Start Vite frontend.
- Trigger a mock stock-report task.
- Confirm real-time log display.
- Confirm generated artifact link appears.

Existing checks must remain green:

```bash

conda run -n ashare-research-lab ruff check .
conda run -n ashare-research-lab pytest -q
```

## Non-Goals

The first UI version will not:

- Provide公网 deployment.
- Add user accounts or RBAC.
- Edit core research YAML configs from the browser.
- Implement automated trading, order management, or broker integration.
- Add buy / sell / target price / position suggestions.
- Replace the existing CLI audit trail.
- Guarantee strict historical PIT research without imported `historical_pit` universe data.

## Open Implementation Notes

- Prefer Server-Sent Events for logs. WebSocket is unnecessary for the first version because the stream is server-to-client.
- Prefer JSON task history over new DuckDB tables for UI runner state. Generated research artifacts still use existing DuckDB audit tables.
- Keep workflow commands explicit and copyable, so any UI run can be reproduced in terminal.
- Do not make the UI infer today. All research dates must be explicit.
